# OneMem V3 Plan

V3 focuses on advanced memory intelligence after V2 has improved evaluation, consolidation quality, embeddings, write hygiene, and temporal retrieval. V3 should not replace the V1/V2 architecture. It should strengthen the same core rule: canonical files are truth, sidecars are rebuildable views, and writes are controlled.

## Primary Goals

- Add LLM-assisted memory operations with deterministic apply.
- Answer reasoning questions using retrieved evidence.
- Improve long-term stability through merge, dedupe, and feedback.
- Add layered summaries without making summaries the truth.
- Improve graph retrieval without introducing unnecessary graph infrastructure.

## 1. LLM-Assisted Consolidator With Controlled Apply

V3 should add an LLM-based consolidator that proposes explicit operations:

- `ADD`
- `UPDATE`
- `MERGE`
- `INVALIDATE`
- `LINK`
- `PROMOTE`
- `DEMOTE`

Important boundary:

- The LLM proposes operations.
- A deterministic memory manager validates and applies operations.
- The chatbot or agent still never writes canonical long-term memory directly.

Validation should check:

- required provenance
- valid node IDs
- valid relation targets
- no silent overwrite of deprecated facts
- confidence and status constraints

## 2. Reasoning Reader

Retrieval alone is not enough for LongMemEval-style questions.

V3 should add a reader step:

```text
query -> retrieve evidence -> reader answers from evidence -> cite memory IDs
```

The reader should support:

- temporal comparisons
- multi-hop questions
- simple counting
- "which happened first" questions
- abstention when evidence is missing

It must cite:

- fact IDs
- episode IDs
- source refs

The reader output should not become canonical memory unless separately captured and consolidated.

## 3. Advanced Dedupe And Merge Queue

V3 should reduce duplicate or near-duplicate long-term facts.

Examples:

```text
Nezir prefers short answers.
Nezir likes concise technical replies.
```

Expected behavior:

- detect as likely duplicates
- keep one canonical fact
- merge evidence refs
- preserve old node as merged/deprecated or alias
- keep audit trail

Initial implementation:

- exact normalized match
- same concept + high embedding similarity
- overlapping source/entity refs
- optional LLM merge suggestion

## 4. Summary Layers

V1 has a single current summary. V3 should add layered summaries:

- session summaries
- topic summaries
- user profile summaries
- project summaries
- archived cluster summaries

Rules:

- Summaries are derived context helpers.
- Facts remain the leading truth layer.
- Summaries must cite source facts or episodes.
- Summaries should be rebuildable or refreshable.

## 5. Graph Reranking

V3 can improve graph retrieval while still avoiding an early graph database.

Add:

- typed relation taxonomy
- relation weights
- concept-neighbor expansion
- hub-control for noisy concepts
- optional personalized PageRank-style scoring

Do not add:

- a dedicated graph database unless SQLite edges are proven insufficient
- trainable graph weights before eval data justifies it

## 6. Abstention And OOD Detection

The system should know when memory is not enough.

V3 should detect:

- no relevant memory found
- weak evidence only
- conflicting evidence
- answer requires reasoning beyond retrieved context

Expected behavior:

```text
I do not have a reliable memory for that.
```

or:

```text
I found related memories, but they do not answer the question directly.
```

## 7. Feedback-Driven Memory Evolution

Memory quality should improve through use.

Signals:

- retrieved and used successfully
- user confirms memory
- user corrects memory
- answer failed due to missing memory
- stale memory never retrieved

Actions:

- increase salience
- promote stable facts to core
- create merge candidates
- invalidate contradicted facts
- lower rank of unused low-salience candidates

Keep it simple:

- Start with explicit feedback commands or API calls.
- Do not add RL or trainable policies in early V3.

## 8. Inspector UI Or Local Review Tool

V3 should add a small inspection layer.

Useful actions:

- list nodes
- inspect a node
- pin/unpin
- invalidate
- view provenance
- view graph neighbors
- approve merge
- rebuild sidecar

This can start as CLI commands before becoming a web UI.

Example:

```bash
onemem inspect fact_123
onemem graph neighbors fact_123
onemem merge approve merge_456
```

## 9. External Benchmark Track

V3 should run a serious benchmark loop.

Benchmarks:

- LongMemEval evidence recall
- LongMemEval answer generation with reader
- LoCoMo subset for long conversation memory
- local golden evals for architecture invariants

Metrics:

- Evidence Recall@5 / Recall@10
- MRR
- answer accuracy with LLM judge
- abstention precision
- context pollution rate
- rebuild consistency
- invalidation correctness

## V3 Success Criteria

- The system can answer a meaningful subset of LongMemEval temporal and multi-hop questions using retrieved evidence.
- Memory writes are operation-based and auditable.
- Duplicate facts are merged without losing provenance.
- Summaries improve context packing without becoming the truth layer.
- The system can abstain when memory is insufficient.
- A human can inspect and correct memory state without editing raw SQLite.

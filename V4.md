# OneMem V4 Plan

V4 collects the items GOAL.md explicitly labels as "nice to have" or "V3-Material that is far from a robust product". They are not required to call the core architecture complete — after V3 the hybrid memory system described in GOAL.md §4 and §11 is already in place. V4 exists so these topics are documented and scoped, not forgotten.

V4 must continue to respect the V1-V3 invariants: Markdown files are truth, sidecars are rebuildable, writes go through a controlled manager, and the agent never mutates canonical long-term memory directly.

## Primary Goals

- Close the small operational gaps V3 left open (boundary triggers, real namespaces, result cache).
- Explore the advanced retrieval and graph ideas from the papers without turning the project into a research codebase.
- Add the richer temporal and modality features only when an eval on existing golden datasets shows they measurably help.
- Keep every addition optional and behind a flag. Hash embeddings and the deterministic path must still work offline.

## 1. Boundary Detection For Consolidation

V1-V3 flush on manual call or after every chat turn. GOAL §7 expects a real boundary trigger: topic change, task completion, session window close, or a large episodic buffer.

V4 should add a `BoundaryDetector` that reads the current episodic buffer and decides whether to flush:

- topic drift, measured as cosine distance between the running topic centroid and the newest episode embedding
- explicit end markers ("task done", "let's move on", `/flush`)
- wall-clock windows (configurable, default 15 minutes of inactivity)
- buffer size threshold (configurable, default N episodes or K tokens)

Rules:

- The detector proposes a boundary. The runtime applies the flush.
- A boundary must never merge episodes across different `session` values.
- Manual flush stays available and wins over the detector.

## 2. Namespace Partitioning

GOAL §11 describes "single writer queue pro user/projekt/session namespace". Today the `session` field is a tag, not a partition. Cross-user or cross-project stores share the same Markdown tree and sidecar.

V4 should introduce first-class namespaces:

- `memory/<namespace>/episodes|facts|concepts|summaries|archive`
- one sidecar per namespace (`memory/<namespace>/.sidecar/index.sqlite3`)
- CLI and API gain an explicit `--namespace` / `namespace=` argument
- retrieval never crosses namespaces unless `include_namespaces=[...]` is passed explicitly

Writer discipline:

- one writer queue per namespace, implemented as a simple process-local lock plus an on-disk lockfile for cross-process safety
- the existing single-runtime path stays the default when no namespace is given (back-compat)

## 3. Retrieval Result Cache

V1-V3 cache embeddings per text; we do not cache retrieval results. GOAL §5 lists a retrieval cache as optional sidecar content.

V4 should add:

- a small LRU cache keyed by `(namespace, query_hash, reference_date, limit, include_hypotheses)`
- invalidated on any write/flush/maintain/invalidate in the namespace
- opt-in via env (`ONEMEM_RETRIEVAL_CACHE=1`) and per-call (`runtime.retrieve(..., cache=False)`)

Rules:

- never serve a stale result after canonical memory has changed
- never store full memory bodies in the cache; store only ranked IDs + scores and re-fetch rows on hit

## 4. Community And Global Summaries

V3 §4 added session / topic / user / project summaries. V4 should add summaries that cut across clusters, not across sessions:

- concept-community summary: a summary of the facts grouped under a tightly-connected concept cluster
- global profile summary: a short, stable summary of the user/system at the namespace level
- archived-cluster summary: a compressed view of what was archived so retrieval can surface "we used to know X"

Rules:

- summaries never become truth; facts and episodes stay the leading layer
- every summary carries `source_refs` pointing to the facts or episodes it compresses
- community detection uses the existing edge table; no new graph backend

## 5. Personalized PageRank And Advanced Graph Reranking

V3 §5 mentioned PPR as optional. V4 should make it real for namespaces where the graph is dense enough to matter.

Scope:

- seed-restart PPR over the concept-and-relation graph, run at retrieval time on the candidate pool only (not globally)
- hub control: down-weight concepts with degree > K (prevents entity hubs, per GAAMA)
- typed-edge weights: distinguish `mentions_concept`, `summarizes`, `contradicts`, `supports`, `alias_of`, `derived_from`

Do not add:

- a dedicated graph database
- anything that breaks sidecar rebuild-from-Markdown

## 6. Bi-Temporal Model

V1-V3 track `valid_from` / `valid_to`. Zep shows that for real-world contradiction handling you also want `transaction_time`: when a fact was recorded, independent of when it was true.

V4 should add:

- `recorded_at` (already implicitly `created_at`) formally promoted to transaction_time
- `valid_from` / `valid_to` kept as event-time fields (already the case after V2)
- retrieval can target either axis: "what did I know on date X" (transaction time) vs. "what was true on date X" (valid time)

Rules:

- changing a fact must update `transaction_time`, not `valid_from`
- temporal rerank (V2) continues to use event-time by default; transaction-time is an opt-in query mode

## 7. Feedback-Learned Retrieval And Write Policies

V3 §7 added explicit feedback signals with deterministic actions. V4 can carefully add utility-learned adjustments, still behind a flag:

- a local per-namespace weights file for the score components in `retrieval.py`
- offline fit from accumulated success/failure feedback
- never a live RL loop; fit is a user-triggered job (`onemem tune`)

Explicit non-goals:

- no gradient-based or RL-trained retrieval policy at inference time
- no learned write policy that decides capture silently; the rule-based write gate remains the authority

## 8. Multimodal Episodes

V1-V3 are text-only. V4 can accept images and audio as episodes:

- episode body stays a textual description; the raw asset lives in `memory/<namespace>/episodes/assets/`
- an `asset_refs` field on the node points to the asset paths
- embedding provider must support multimodal input (Gemini Embedding multimodal is one option) — hash fallback stays text-only
- retrieval stays text-first; multimodal similarity is an optional second pass

Rules:

- assets are not canonical truth; the Markdown body and frontmatter remain authoritative
- no OCR/transcription inside the hot write path; those run asynchronously and update the node body

## 9. Agent-Native Sandboxed Curation

ByteRover describes agent-driven memory curation inside a sandboxed code environment. V4 can add a much smaller version:

- a `curate` CLI that spawns a restricted subprocess with read-only access to the Markdown tree
- the subprocess may propose `MERGE` / `INVALIDATE` / `LINK` operations, same schema as V3 §1
- operations run through the same deterministic apply step as V3 — no free-form edits

Scope limit:

- no general code execution environment
- no tool-calling harness
- this is only a proposer, not a writer

## 10. Strategy And Meta-Memory

GOAL §12 mentions strategy/meta-memory "only if the agent should learn from task trajectories". V4 should keep this minimal:

- a `trajectory` episode kind storing successful or failed task runs, with outcome and cost
- aggregation into `strategy` facts: "for task X, prefer approach Y because it succeeded on N similar trajectories"
- retrieval of strategy facts only when the user or agent opts in (e.g., `include_strategy=True`)

Explicit non-goals:

- no self-reflection loop
- no automatic strategy generation without evidence
- strategies count as facts and follow the same lifecycle (ephemeral → candidate → stable → core)

## 11. Inspector UI

V3 §8 listed CLI inspection commands. V4 can add a minimal local web UI:

- read-only by default; pin/unpin/invalidate/approve-merge require an explicit `--write` flag at startup
- served from the local machine, never exposed remotely
- reuses the existing `MemoryRuntime` and `MarkdownStore`; no background process needed
- optional `onemem serve --port 7070`

## 12. External Benchmark Automation

V3 §9 defined the benchmark scope. V4 should make it automatic:

- nightly local run against `evals/`
- optional LongMemEval and LoCoMo slices
- track metric deltas vs. the previous run
- surface regressions before they ship

Rules:

- benchmarks must work with the hash provider so CI is free
- Gemini-backed runs stay opt-in and rate-limited

## V4 Success Criteria

- A fresh project can choose a namespace and not pollute other namespaces.
- Flush happens at real boundaries, not on every chat turn.
- Retrieval is noticeably faster on repeated queries thanks to the result cache.
- PPR-based graph reranking improves evidence recall on at least one eval slice, or is removed.
- A user can ask "what did I know on date X" and get a bi-temporal answer.
- Multimodal episodes can be captured and later retrieved without breaking the Markdown-first invariant.
- No V4 feature requires giving up sidecar rebuild or offline hash-embedding operation.

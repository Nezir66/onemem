V2 focuses on quality, memory hygiene, and better evaluation. The goal is not a larger platform, but a stronger version of the current V1 architecture: file-backed canonical memory, rebuildable sidecars, episodic buffer, controlled consolidation, and inspectable retrieval.

## Primary Goals

- Improve consolidation quality without losing auditability.
- Reduce memory spam before it reaches long-term memory.
- Make retrieval more semantic and more explainable.
- Add temporal/event awareness for real long-term memory questions.
- Upgrade evals from simple answer-string matching to evidence recall.

## 1. Evidence-Recall Eval For LongMemEval

Current LongMemEval import checks whether the final answer text appears in retrieved context. This is too strict for temporal and reasoning questions where the answer must be derived.

V2 should measure:

- whether the correct evidence session is retrieved in Top K
- rank of the first correct evidence session
- Recall@5 and Recall@10
- MRR over evidence sessions
- optional answer-string checks only for direct extraction questions

Implementation direction:

- Preserve LongMemEval metadata in eval cases.
- Add `expected_source_refs` or `expected_session_ids`.
- Let `EvalRunner` score evidence hits separately from answer-text hits.
- Keep the existing simple eval format for local golden tests.

## 2. Better Consolidation

The current consolidator splits text into sentences and derives concepts from frequent tokens. This is useful for V1, but weak for production memory.

V2 should extract:

- atomic facts
- user preferences
- project decisions
- events
- dates and relative time references
- entities and aliases
- source evidence
- confidence and salience

Important constraints:

- The consolidator must still write through `MemoryRuntime` / `MarkdownStore`.
- Canonical Markdown remains the source of truth.
- No direct mutation by the chatbot or agent.
- LLM-assisted extraction is allowed, but the apply step must remain deterministic.

Suggested first version:

- Improve deterministic extraction before adding an LLM.
- Add better multilingual stopwords.
- Detect dates with simple patterns.
- Classify facts into lightweight categories using rules.
- Add a later `LLMConsolidator` behind the same interface.

## 3. Memory-Write Filter

The chatbot currently captures every interaction. This creates memory spam.

V2 should add a write gate that decides:

- should this turn be stored?
- should it stay episodic only?
- should it be considered for consolidation?
- what salience should it get?
- does it need a TTL?

Initial rule-based policy:

- Store explicit memory requests: "remember", "merk dir", "keep in mind".
- Store stable preferences and project decisions.
- Store corrections and contradictions.
- Ignore greetings, acknowledgements, and transient chatter.
- Lower salience for low-information turns.

Later LLM policy:

- The LLM may suggest `capture`, `ignore`, or `capture_low_salience`.
- The runtime applies the decision, not the LLM directly.

## 4. Real Embedding Provider

The current vector layer uses deterministic hashing vectors. It is offline and rebuildable, but not deeply semantic.

V2 should introduce an embedding provider abstraction:

- `HashEmbeddingProvider` as default offline fallback
- Gemini Embedding provider as the planned first real provider
- optional local embedding model later

Rules:

- Embeddings remain derived sidecar data.
- Rebuild from Markdown must still work.
- The provider name and embedding dimension should be stored in sidecar metadata.
- Retrieval must still work without network access.

### Planned Provider: Gemini Embedding 2

For V2, the preferred hosted embedding provider should be Google's Gemini Embedding line, with Gemini Embedding 2 as the target if it is available and appropriate at implementation time.

Before implementation, check the current official Google documentation:

- Gemini API embeddings docs
- Vertex AI embeddings docs
- Gemini Embedding 2 model page / release notes
- current model IDs, dimensions, token limits, pricing, preview/GA status, and rate limits

Current note as of April 2026:

- Google documents `gemini-embedding-001` as the stable text embedding model in the Gemini API docs.
- Google has announced Gemini Embedding 2 as a public-preview multimodal embedding model.
- V2 should not hardcode assumptions from old examples. The implementation must verify the latest model ID and request shape from Google's current docs before coding.

Implementation target:

- Add a `GeminiEmbeddingProvider`.
- Keep `HashEmbeddingProvider` as offline fallback.
- Store provider metadata in the sidecar.
- Allow configurable output dimensionality when supported.
- Prefer text-only usage first; multimodal support can remain a later extension unless the V2 scope explicitly needs it.

## 5. Better Concepts And Aliases

Current concepts can be noisy, e.g. common words become concept anchors.

V2 should improve:

- English and German stopwords
- canonical concept IDs
- alias register
- simple entity/concept distinction
- concept dedupe
- topic labels such as `user_profile`, `answer_style`, `project_architecture`

Expected result:

- fewer junk concepts
- stronger graph expansion
- more useful inspection in Markdown

## 6. Temporal Retrieval

> Done. See `src/onemem/temporal.py`, `src/onemem/retrieval.py::TEMPORAL_WEIGHT`,
> `MemoryRuntime.capture(..., event_date=...)`, `MemoryRuntime.retrieve(..., reference_date=...)`,
> `LongMemEvalImporter._convert_item`, and `tests/test_temporal.py`. Dates are
> parsed deterministically (regex only, no calendar library); temporal queries
> get a bounded +0.12 score term; non-temporal queries are unchanged.

LongMemEval shows that many useful memory questions are temporal.

V2 should add:

- date extraction from episodes
- `event_date` or stronger use of `valid_from`
- reference-date handling for eval queries
- query detection for "first", "last", "before", "after", "how many days"
- temporal reranking

Scope limit:

- V2 should not build a full temporal reasoning engine.
- It should improve retrieval of the right evidence, not always compute final answers.

## 7. Retrieval Debug Output

V2 should make retrieval decisions inspectable.

Add an optional debug mode showing:

- lexical score
- vector score
- graph boost
- salience
- confidence
- recency score
- final score
- whether the result came from seed search or graph expansion

CLI target:

```bash
onemem retrieve "query" --debug
```

Eval target:

```bash
onemem eval run evals/foo.json --include-context --json
```

## 8. Maintenance Improvements

> candidate decay, hypothesis TTL, deprecated filtering) and the archive bug fix in
> `src/onemem/markdown_store.py::MarkdownStore.archive` that now persists the

Improve existing maintenance without adding a complex scheduler.

V2 should support:

- archive low-salience stale episodes
- decay unused candidate facts
- expire unconfirmed hypotheses
- keep pinned/core nodes
- rebuild sidecar after maintenance

Add tests for:

- pinned nodes are not archived
- deprecated nodes are filtered from retrieval
- archived episodes are not indexed as active memory

- Chatbot no longer stores every trivial interaction.
- Concepts are visibly cleaner in Markdown.
